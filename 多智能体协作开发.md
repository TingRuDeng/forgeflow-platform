# 多智能体协作开发 v1 操作手册

## 1. 目标

本手册用于落地第一版多智能体协作开发系统，目标是：

- 你只和一个总控制智能体交互
- 总控制智能体负责需求澄清、任务拆分、任务分发、PR 审查与合并决策
- 多个执行型智能体分别处理后端、重构、测试、工具链与前端任务
- GitHub 作为唯一真相源
- 第一版优先追求**可预测、可审计、可回放、可恢复**
- 第一版**不追求完全自动化抢单**

---

## 2. 第一版最终架构

```text
你
  ↓
codex-control (GPT-5.4)
  ├─ 需求澄清
  ├─ 任务拆分
  ├─ 任务归类（codex-pool / gemini-pool）
  ├─ 审查 PR
  └─ 决定 merge / rework

dispatcher (Node/TS)
  ├─ 任务分解结果消费
  ├─ provider 能力表 / doctor 预检
  ├─ 任务状态机
  ├─ worker 注册与心跳
  ├─ worktree 生命周期管理
  ├─ 从池中挑空闲 worker
  ├─ 记录任务、PR、重试
  └─ 输出观测事件

worker pools
  ├─ codex-worker-*  (GPT-5.2-Codex)
  └─ gemini-worker-* (gemini-2.5-pro)

GitHub
  ├─ repo / issues / PRs
  ├─ Actions / CI
  ├─ protected branch
  └─ required checks / required reviews
```

### 2.1 当前默认落地方式

为了先把系统跑通，当前默认采用最简模式：

- 单个源仓库
- 多任务分支
- 多 `git worktree`
- 所有 worker 都直接在源仓库的任务分支上工作
- 最后统一通过 PR 合并到 `main`

当前明确不做：

- 每个 worker 一个 GitHub 账号
- 每个 worker 一个 fork
- 把复杂权限治理作为 v1 前置条件

## 3. 第一版角色定义

### 3.1 codex-control（唯一实例）

模型：`GPT-5.4`

职责：

- 与你交互
- 澄清需求
- 拆分任务
- 给任务选择池子：
  - `codex`
  - `gemini`
- 生成任务约束：
  - `allowed paths`
  - `depends_on`
  - `acceptance`
  - `priority`
  - `verification_mode`
- 审查 PR
- 决定：
  - approve
  - request changes
  - reassign
  - merge

> 第一版里 `codex-planner` 和 `codex-integrator` **先合并成一个实例**，统一叫 `codex-control`。

### 3.2 codex-worker（多个实例）

模型：`GPT-5.2-Codex`

职责：

- 后端开发
- 重构
- 测试补充
- 工具链脚本
- CI 修复
- 非前端主视觉类任务

适合目录：

- `apps/api/**`
- `services/**`
- `tooling/**`
- `scripts/**`
- `packages/shared/**`

### 3.3 gemini-worker（多个实例）

模型：`gemini-2.5-pro`

职责：

- Vue 前端页面
- 组件开发
- 路由与状态管理
- 表单、交互、页面逻辑
- 前端测试

适合目录：

- `apps/web/**`

### 3.4 dispatcher（系统服务）

实现语言：Node.js / TypeScript

职责：

- 保存任务状态
- 保存 worker 注册信息
- 记录心跳
- 给指定池挑选空闲 worker
- 下发任务
- 收集执行结果
- 处理失败与重试

> dispatcher **不负责理解需求**，只负责执行调度规则。

### 3.5 借鉴 `iccc` 后新增的 v1 机制

- 每个任务强制独立 `git worktree`
- 增加结构化任务分解器，先拆后派
- 增加轻量交叉验证链路：
  - worker 自测
  - control/reviewer 复审
  - GitHub CI 硬门禁
- 增加最小观测面：
  - worker 状态
- 最近任务事件
- 最近 PR/CI 状态

### 3.6 借鉴 `mco` 后新增的 v1 机制

- 增加 provider 能力表：
  - 哪些 CLI 可用
  - 支持哪些权限键
  - 默认运行参数
- 增加 `doctor` 预检：
  - 检查 CLI binary
  - 检查认证状态
  - 检查项目契约与 provider 能力是否冲突
- 明确区分两类模式：
  - `run`
  - `review`
- 增加版本化结果契约：
  - `run-result`
  - `review-findings`
- 运行态增加：
  - `retry`
  - `partial_success`
  - `expired`

### 3.7 当前不吸收 `mco` 的部分

- 不把“同一任务 fan-out 给多个 provider”作为默认开发模式
- 不让 MCO 风格的 neutral CLI 替代 `codex-control` 成为你的主入口
- 不把跨会话 memory bridge 作为 v1 基础能力

### 3.8 `mco` 风格但放到 Phase 2 的部分

- 同题多 provider 并行 review
- findings 去重和 `detected_by` 溯源
- synthesis 总结层
- SARIF 导出
- 历史 findings / agent scoring / memory 注入

## 4. 第一版核心原则

### 4.1 GitHub 是唯一真相源

所有这些内容最终都要落到 GitHub 或本地持久化中：

- Epic / 需求
- 任务
- PR
- Review
- CI 状态
- Merge 结果

### 4.2 第一版不做自动抢单

第一版不采用：

- worker 自主抢任务
- 多 worker 自主协商分工
- 无中心分发

第一版采用：

- `codex-control` 先判断任务归属池
- dispatcher 先运行 `doctor` 预检
- dispatcher 再从该池中选一个空闲 worker
- dispatcher 先为任务创建独立 worktree
- worker 只执行被明确分配给自己的任务

### 4.3 第一版先不把 Trae / OpenClaw 放进主链路

第一版自动化主链路只使用：

- `codex-control`
- `codex-worker`
- `gemini-worker`
- `dispatcher`
- `GitHub`

后续再扩展：

- Trae CLI
- OpenClaw
- ACP 会话控制层

### 4.4 第一版不使用 fork 模式

当前第一版统一使用：

- 源仓库任务分支
- 独立 worktree
- PR 合并

不使用：

- worker 自有 fork
- fork 到上游的多跳同步
- 每个 worker 自己维护一份远端仓库

## 6. GitHub 侧配置

### 6.1 分支保护

给 `main` 开启以下限制：

- 必须通过 PR 合并
- 必须通过 required status checks
- 必须有 review
- 禁止直接 push
- 禁止 force push
- 有条件再开启 merge queue

### 6.2 self-hosted runner

建议第一版至少准备 1 个 self-hosted runner。

用途：

- 跑 CI
- 跑 dispatcher 兜底 workflow
- 执行自动化任务

### 6.3 GitHub Workflow 入口

第一版保留两个入口：

- 主入口：你直接和 `codex-control` 对话，由它调用 dispatcher
- 备用入口：GitHub `workflow_dispatch`

### 7.1 `AGENTS.md`

```md
# Team rules

## Global
- Never push directly to main.
- Always work through task branch + PR.
- Only modify files listed in the assigned task.

- For backend changes, run the package-local lint/test/build commands.
- Include test evidence in the PR body.

- Branch: ai/<pool>/<task-id>-<slug>
- Commit style: feat/fix/refactor/test/chore(scope): summary

- Stack: Vue 3 + TypeScript + Vite + Pinia + Vue Router
- UI tasks only under apps/web/**

- Prefer composition API
- Keep components small and strongly typed
- Add or update Playwright / Vitest coverage when changing user flows
```

## 10. Dispatcher 技术选型

第一版建议：

- Node 20
- Fastify
- SQLite
- Octokit
- Zod

### 10.1 `workers` 数据结构

```ts
type WorkerPool = "codex" | "gemini";

interface Worker {
  id: string;
  pool: WorkerPool;
  host: string;
  model: string;
  status: "idle" | "busy" | "offline";
  lastHeartbeatAt: string;
  currentTaskId?: string;
}
```

### 10.2 `tasks` 数据结构

```ts
type TaskStatus =
  | "planned"
  | "ready"
  | "assigned"
  | "in_progress"
  | "review"
  | "merged"
  | "blocked"
  | "failed";

interface Task {
  id: string;
  epicId: string;
  title: string;
  pool: "codex" | "gemini";
  paths: string[];
  dependsOn: string[];
  acceptance: string[];
  branchName: string;
  status: TaskStatus;
  assigneeWorkerId?: string;
  prNumber?: number;
}
```

### 10.3 第一版分发策略

- `codex-control` 决定任务属于哪个池
- dispatcher 从该池中挑一个空闲 worker
- worker 不自动抢单

池内选择算法：

1. 过滤 `status = idle`
2. 选空闲时间最长的
3. 并列时 round-robin

## 15. 日常使用流程

### 15.1 你如何发需求

直接对 `codex-control` 说：

```text
需求：
给管理后台增加登录、角色权限、菜单控制、审计日志。
前端 Vue 3 + TS，后端 Node。
先澄清需求，拆任务，不要直接写代码。
```

### 15.2 codex-control 的职责

它必须先输出：

- 需求摘要
- 缺失信息
- 假设
- 任务树
- 每个任务的 pool
- `allowed paths`
- `acceptance`
- `merge order`

### 15.3 你确认后

你回复：

```text
确认计划，开始执行
```

然后 dispatcher 开始分发：

- 前端任务 → `gemini-worker-*`
- 后端 / 重构 / 测试 / 工具链 → `codex-worker-*`

### 15.4 PR 回来后

仍由 `codex-control` 进入 integration mode：

- 看 diff
- 看 CI
- 判断是否满足 acceptance
- 决定：
  - approve
  - request changes
  - reassign
  - merge

## 17. 失败恢复策略

### 17.1 worker 挂掉

- dispatcher 根据 `lastHeartbeatAt` 把 worker 标记为 `offline`
- task 回滚到 `ready`
- 由 `codex-control` 决定是否重派

### 17.2 PR 被驳回

- task 状态改成 `blocked`
- integrator 写明原因
- planner 重新生成修复任务或原任务 retry

### 17.3 路径冲突

- 同一主路径块只允许一个任务持有写锁
- 例如：`apps/web/auth/**` 同时只能有一个 `in_progress` task
